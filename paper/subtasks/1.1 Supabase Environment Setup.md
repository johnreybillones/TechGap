## **🛠️ Subtask 1.1: Supabase Environment Setup**

### **1\. Enable pgvector Extension**

Before creating any tables, the vector extension must be activated. This allows PostgreSQL to handle the 384-dimensional embeddings generated by the SBERT model later in Phase 2\.

**Action:** Open the Supabase SQL Editor and run:

SQL  
\-- Enable the pgvector extension to work with embeddings  
CREATE EXTENSION IF NOT EXISTS vector;

---

### **2\. Database Schema Definition**

The developers should execute the following schema. This design ensures we store the raw data for auditing while preparing a "clean" space for the machine-learning results.

#### **A. Raw Jobs Table (jobs\_raw)**

This table acts as your "Landing Zone." We save the entire JSON response from Adzuna and the scrapers here to ensure we never lose data.

SQL  
\-- A. Raw Jobs Table (jobs\_raw)  
CREATE TABLE jobs\_raw (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    source TEXT NOT NULL,          \-- 'adzuna', 'indeed', 'jobstreet'  
    external\_id TEXT UNIQUE,         
    job\_title TEXT,                \-- The role name  
    contract\_type TEXT,            \-- Added: 'Full-time', 'Contract'  
    seniority\_level TEXT,          \-- Added: 'Junior', 'Senior', 'Lead'  
    company\_name TEXT,  
    location TEXT,  
    raw\_json JSONB,                  
    created\_at TIMESTAMPTZ DEFAULT NOW()  
);

#### **B. Curriculum Tracks Table (curriculum\_tracks)**

This table stores the "Target" data—the academic requirements we are analyzing.

SQL  
\-- B. Curriculum Tracks Table (curriculum\_tracks)  
CREATE TABLE curriculum\_tracks (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    track\_code TEXT NOT NULL,      \-- 'CS-IS', 'CS-GD', 'IT-WD', 'IT-NT'  
    track\_name TEXT NOT NULL,  
    course\_code TEXT,                
    course\_title TEXT,  
    outcome\_text TEXT,               
    embedding VECTOR(384),  
    hybrid\_embedding VECTOR(448),  \-- Added: \[SBERT (384) \+ GCN (64)\]  
    bloom\_distribution JSONB,      \-- Added: Stores heatmap data for dashboard       
    created\_at TIMESTAMPTZ DEFAULT NOW()  
);

#### **C. Processed Skills Library (skills\_library)**

This is where the **Inference Engine** will live. It links specific skills to their "Parents" (e.g., React $\\rightarrow$ JavaScript).

SQL  
CREATE TABLE skills\_library (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    skill\_name TEXT UNIQUE NOT NULL,  
    parent\_skill\_id UUID REFERENCES skills\_library(id), \-- For the Inference Engine  
    category TEXT,                 \-- 'Frontend', 'Cloud', 'Data Science', etc.  
    industry\_demand\_weight FLOAT DEFAULT 0.0,           \-- For Adaptive Ranking  
    bloom\_level SMALLINT CHECK (bloom\_level BETWEEN 1 AND 6), \-- Added  
    embedding VECTOR(384),         \-- Raw SBERT  
    graph\_embedding VECTOR(64),    \-- Added: GCN/Node2Vec output  
    hybrid\_embedding VECTOR(448)   \-- Added: Combined vector for Inference Engine  
);

Add this SQL command at the very end of your schema script to enable the "lightning-fast" semantic search mentioned in your technical basis:

SQL  
\-- Add HNSW index for the 448-dimensional hybrid embeddings  
CREATE INDEX ON job\_embeddings USING hnsw (embedding vector\_cosine\_ops)  
WITH (m \= 16, ef\_construction \= 64);

### **4\. Why this schema matters for the Vision**

* **Auditability:** By using JSONB in jobs\_raw, if the model ever hallucinates a skill, we can go back and see the original text the scraper found.  
* **Speed:** Placing the embedding directly in the curriculum\_tracks table allows for "Nearest Neighbor" searches directly in SQL, meaning the "Run Analysis" button will be nearly instant for the user.  
* **Inference Readiness:** The parent\_skill\_id in the skills\_library table allows developers to build the **Inference Engine** logic (e.g., "Find all jobs that require React OR its parent JavaScript").
