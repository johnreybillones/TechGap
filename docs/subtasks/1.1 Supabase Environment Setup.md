To get the project moving, the developers must first establish a robust and scalable data foundation. Since we are moving to **Supabase** and utilizing **pgvector** for semantic similarity, the database schema must be more than just a storage bin; it must be optimized for high-dimensional vector math.

> **Architecture note:** Curriculum data (syllabus PDFs parsed into structured CSV) is **not stored in Supabase**. It is loaded in-memory by the Python pipeline at inference time. Only raw job data and **model output results** are persisted in Supabase — this keeps the database lean and well within the free-tier limits.

Here is the step-by-step implementation guide for **Subtask 1.1: Supabase Environment Setup.**

---

## **🛠️ Subtask 1.1: Supabase Environment Setup**

### **1\. Enable pgvector Extension**

Before creating any tables, the vector extension must be activated. This allows PostgreSQL to handle the 384-dimensional embeddings generated by the SBERT model later in Phase 2\.

**Action:** Open the Supabase SQL Editor and run:

SQL  
\-- Enable the pgvector extension to work with embeddings  
CREATE EXTENSION IF NOT EXISTS vector;

---

### **2\. Database Schema Definition**

The developers should execute the following schema in order. The design separates job data (persistent, grows over time) from model outputs (written after each analysis run).

#### **A. Raw Jobs Table (jobs\_raw)**

This table acts as your "Landing Zone." We save the entire JSON response from Adzuna and the scrapers here to ensure we never lose data.

SQL  
\-- A. Raw Jobs Table (jobs\_raw)  
CREATE TABLE jobs_raw (  
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),  
    source TEXT NOT NULL,          -- 'adzuna', 'jobstreet'  
    external_id TEXT,              -- Deduplication key from source  
    job_title TEXT,                -- The role name  
    description TEXT,              -- Raw job description (source for SBERT embeddings)  
    contract_type TEXT,            -- 'Full-time', 'Contract', 'Part-time'  
    seniority_level TEXT,          -- 'Junior', 'Senior', 'Lead' (NULL until Week 2 NLP)  
    company_name TEXT,  
    location TEXT,                 -- City/region in Philippines  
    classification TEXT,           -- ICT sub-category (e.g., 'Information Technology')  
    salary_min FLOAT,              -- Minimum salary (from API/scraper)  
    salary_max FLOAT,              -- Maximum salary (from API/scraper)  
    salary_currency TEXT DEFAULT 'PHP',  
    track_alignment TEXT,          -- Assigned track after NLP: 'CS-IS', 'CS-GD', 'IT-WD', 'IT-NT' (NULL until Week 2)  
    source_url TEXT,               -- Direct link to the original posting (for audit/debug)  
    posted_at TIMESTAMPTZ,         -- When the employer published the listing  
    scraped_at TIMESTAMPTZ DEFAULT NOW(), -- When we collected it  
    description_embedding VECTOR(384), -- SBERT embedding on description text (NULL until Week 2)  
    raw_json JSONB,                -- Full original response for audit/re-processing  
    created_at TIMESTAMPTZ DEFAULT NOW(),  
    UNIQUE (source, external_id)   -- Composite dedup key: one row per (source, job ID)  
);  
  
\-- Scope constraint: Only accept posts within the last 6 months  
\-- Enforced in scraper logic — do not ingest posted\_at older than 180 days.

#### **B. Job Skills Junction Table (job\_skills)**

Normalized many-to-many: one row per (job, skill) pair. Populated in Week 2 NLP pipeline after skill extraction.

SQL  
CREATE TABLE job\_skills (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    job\_id UUID NOT NULL REFERENCES jobs\_raw(id) ON DELETE CASCADE,  
    skill\_id UUID NOT NULL REFERENCES skills\_library(id) ON DELETE CASCADE,  
    frequency INT DEFAULT 1,      \-- How many times this skill appeared in the description  
    UNIQUE(job\_id, skill\_id)  
);

#### **C. Processed Skills Library (skills\_library)**

This is where the **Inference Engine** lives. It links specific skills to their "Parents" (e.g., React → JavaScript) and stores the pre-curated topic breakdowns used for deterministic output generation.

SQL  
CREATE TABLE skills\_library (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    skill\_name TEXT UNIQUE NOT NULL,  
    parent\_skill\_id UUID REFERENCES skills\_library(id), \-- For the Inference Engine  
    category TEXT,                 \-- 'Frontend', 'Cloud', 'Data Science', etc.  
    industry\_demand\_weight FLOAT DEFAULT 0.0,           \-- For Adaptive Ranking  
    bloom\_level SMALLINT CHECK (bloom\_level BETWEEN 1 AND 6),  
    suggested\_topics JSONB,        \-- Pre-curated sub-topic list, e.g. {"topics": ["Dockerfile syntax", "Docker Compose", "K8s basics"]}  
    program\_outcomes TEXT[],       \-- CHED outcome codes, e.g. ARRAY['IT03', 'IT07']  
    embedding VECTOR(384),         \-- Raw SBERT  
    graph\_embedding VECTOR(64),    \-- Node2Vec output  
    hybrid\_embedding VECTOR(448)   \-- Combined vector for Inference Engine  
);

---

#### **D. Model Output Tables**

These 4 tables store the results of each analysis pipeline run. The frontend dashboard reads exclusively from these tables. They are written once per run and are small (< 1 MB total).

SQL  
\-- D1. Analysis Runs (one row per track per pipeline run)  
CREATE TABLE analysis\_runs (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    track\_code TEXT NOT NULL,           \-- 'IT-WD', 'CS-IS', etc.  
    track\_name TEXT NOT NULL,           \-- Full display name  
    program\_type TEXT NOT NULL,         \-- 'BSCS' or 'BSIT'  
    alignment\_score FLOAT,             \-- 0.0–1.0 overall alignment (Tier 1)  
    skills\_matched INT DEFAULT 0,      \-- Green count  
    skills\_inferred INT DEFAULT 0,     \-- Yellow count (logically covered by parent skills)  
    skills\_missing INT DEFAULT 0,      \-- Red count (true gaps)  
    bloom\_distribution JSONB,          \-- {"L1":3,"L2":5,"L3":2,"L4":0,"L5":0,"L6":0}  
    total\_courses INT,                 \-- Number of courses in this track  
    total\_units INT,                   \-- Sum of credit units (for 146-floor display)  
    jobs\_analyzed INT,                 \-- How many jobs were in the comparison set  
    run\_at TIMESTAMPTZ DEFAULT NOW()  
);  
  
\-- D2. Gap Results (one row per identified gap skill, Tier 2)  
CREATE TABLE gap\_results (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    run\_id UUID NOT NULL REFERENCES analysis\_runs(id) ON DELETE CASCADE,  
    rank INT NOT NULL,                          \-- Position in ranked list  
    skill\_name TEXT NOT NULL,  
    urgency\_score FLOAT,                        \-- XGBRanker output (0–1)  
    industry\_demand\_pct FLOAT,                  \-- e.g., 0.43 = 43%  
    bloom\_level TEXT,                           \-- e.g., 'L3-L4'  
    action\_type TEXT NOT NULL,                  \-- 'MODIFY', 'ELECTIVE', 'NEW_COURSE', 'ENHANCE'  
    anchor\_course\_code TEXT,  
    anchor\_course\_title TEXT,  
    anchor\_similarity FLOAT,  
    course\_type TEXT,                           \-- 'core', 'professional', 'elective', 'GE'  
    is\_ched\_protected BOOLEAN DEFAULT FALSE,  
    topics\_to\_add JSONB,                        \-- From skills_library.suggested_topics  
    replace\_candidate TEXT,                     \-- Module title from CSV (e.g., "Module 6: PHP Web Concepts & MySQL")  
    replace\_candidate\_hours INT,               \-- Hours allocated to the replaced module  
    replace\_candidate\_demand\_pct FLOAT,  
    replace\_rationale TEXT,  
    bloom\_before JSONB,                         \-- {"L1":3,"L2":5,"L3":2,"L4":0}  
    bloom\_after JSONB,  
    alignment\_delta FLOAT,                      \-- e.g., 0.07 = +7pp  
    new\_job\_matches INT,  
    program\_outcomes TEXT[],                    \-- CHED outcome codes  
    alternative\_courses JSONB                   \-- [{"code":"IT312","title":"...","similarity":0.58}, ...]  
);  
  
\-- D3. Obsolescence Results (one row per flagged low-demand topic, Tier 3)  
CREATE TABLE obsolescence\_results (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    run\_id UUID NOT NULL REFERENCES analysis\_runs(id) ON DELETE CASCADE,  
    topic\_name TEXT NOT NULL,                   \-- Module title from CSV  
    current\_course\_code TEXT,  
    current\_course\_title TEXT,  
    industry\_demand\_pct FLOAT,  
    bloom\_level TEXT,  
    suggested\_replacement TEXT,  
    replacement\_demand\_pct FLOAT,  
    demand\_ratio FLOAT,                         \-- replacement / current  
    is\_ched\_protected BOOLEAN DEFAULT FALSE,  
    suggestion TEXT                              \-- Template-generated recommendation text  
);  
  
\-- D4. Job Role Mappings (one row per course, Tier 4)  
CREATE TABLE job\_role\_mappings (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    run\_id UUID NOT NULL REFERENCES analysis\_runs(id) ON DELETE CASCADE,  
    course\_code TEXT NOT NULL,  
    course\_title TEXT NOT NULL,  
    current\_roles JSONB,          \-- [{"role":"Frontend Developer","match_pct":72}, ...]  
    projected\_roles JSONB,        \-- [{"role":"React Developer","match_pct":89,"is_new":true}, ...]  
    new\_roles\_count INT DEFAULT 0,  
    new\_postings\_count INT DEFAULT 0  
);

> **Note:** Tier 5 (Executive Summary Report) has no dedicated table — it is assembled at query time by the FastAPI endpoint aggregating data from all 4 output tables for a given `run_id`.

### **3\. Indexes**

Add these SQL commands after the table creation to enable fast search.

SQL  
\-- HNSW index for skills\_library hybrid embeddings (448-dim)  
\-- Using HNSW because the skills table is small and fixed-size.  
CREATE INDEX ON skills\_library USING hnsw (hybrid\_embedding vector\_cosine\_ops)  
WITH (m \= 16, ef\_construction \= 64);  
  
\-- IVFFlat index for jobs\_raw description search (384-dim, high row count)  
\-- IVFFlat is preferred over HNSW for large, growing datasets like jobs_raw.  
CREATE INDEX ON jobs\_raw USING ivfflat (description\_embedding vector\_cosine\_ops)  
WITH (lists \= 100);  
  
\-- B-tree indexes for common filter queries on jobs_raw  
CREATE INDEX idx\_jobs\_raw\_track ON jobs\_raw (track\_alignment);  
CREATE INDEX idx\_jobs\_raw\_posted ON jobs\_raw (posted\_at DESC);  
CREATE INDEX idx\_jobs\_raw\_source ON jobs\_raw (source);  
  
\-- Junction table indexes for fast per-skill demand computation  
CREATE INDEX idx\_job\_skills\_skill ON job\_skills (skill\_id);  
CREATE INDEX idx\_job\_skills\_job ON job\_skills (job\_id);  
  
\-- Output table indexes for fast dashboard fetching  
CREATE INDEX idx\_gap\_results\_run\_rank ON gap\_results (run\_id, rank);  
CREATE INDEX idx\_obsolescence\_run ON obsolescence\_results (run\_id);  
CREATE INDEX idx\_job\_mappings\_run ON job\_role\_mappings (run\_id);

### **5\. Why this schema matters for the Vision**

* **Curriculum stays lean:** Syllabus data is parsed into CSV and processed in-memory by Python. This eliminates ~10 MB of curriculum embeddings from the database and removes the need for HNSW indexes on curriculum tables — significant savings within the free tier.
* **Output tables are the API layer:** The Next.js frontend reads exclusively from `analysis_runs`, `gap_results`, `obsolescence_results`, and `job_role_mappings`. The FastAPI backend writes to them after each pipeline run. This clean separation means the dashboard is completely decoupled from the ML pipeline.
* **Auditability:** The `raw_json JSONB` in `jobs_raw` means if the model ever hallucinates a skill, we can go back and see the exact original text the scraper found. `source_url` and `posted_at` provide full provenance.
* **Normalization:** `job_skills` is a junction table (many-to-many), not an array column. This enables `GROUP BY skill_id` queries for computing `industry_demand_weight` without array unnesting.
* **Speed:** HNSW on `skills_library` supports sub-millisecond nearest-neighbor searches for the ontology. IVFFlat on `jobs_raw` handles the larger row count efficiently.
* **Scalability:** The 6-month `posted_at` scope constraint keeps `jobs_raw` bounded. The `track_alignment` column allows fast per-track filtering without JOINs during inference.
* **Inference Readiness:** `parent_skill_id` in `skills_library` allows the Inference Engine to traverse the skill graph (e.g., "Find all jobs that require React OR its parent JavaScript"). `suggested_topics` enables specific topic-level recommendations without AI API calls. `program_outcomes` ensures every suggestion maps to a CHED-mandated outcome.
* **Replace Candidate Precision:** Replacement candidates for obsolete topics come from the actual `modules` JSON loaded from CSV (real module titles like "Module 6: PHP Web Concepts & MySQL" with their allocated hours), not from embedding-based guesses. This enables specific, actionable phrasing like "reduce the 6-hour PHP Web Concepts module."
