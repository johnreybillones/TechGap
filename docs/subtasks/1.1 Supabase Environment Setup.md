## **🛠️ Subtask 1.1: Supabase Environment Setup**

### **1\. Enable pgvector Extension**

Before creating any tables, the vector extension must be activated. This allows PostgreSQL to handle the 384-dimensional embeddings generated by the SBERT model later in Phase 2\.

**Action:** Open the Supabase SQL Editor and run:

SQL  
\-- Enable the pgvector extension to work with embeddings  
CREATE EXTENSION IF NOT EXISTS vector;

### **2\. Database Schema Definition**

The developers should execute the following schema. This design ensures we store the raw data for auditing while preparing a "clean" space for the machine-learning results.

#### **A. Raw Jobs Table (jobs\_raw)**

This table acts as your "Landing Zone." We save the entire JSON response from Adzuna and the scrapers here to ensure we never lose data.

SQL  
\-- A. Raw Jobs Table (jobs\_raw)  
CREATE TABLE jobs\_raw (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    source TEXT NOT NULL,          \-- 'adzuna', 'jobstreet'  
    external\_id TEXT UNIQUE,       \-- Deduplication key from source  
    job\_title TEXT,                \-- The role name  
    description TEXT,              \-- Raw job description (source for SBERT embeddings)  
    contract\_type TEXT,            \-- 'Full-time', 'Contract', 'Part-time'  
    seniority\_level TEXT,          \-- 'Junior', 'Senior', 'Lead' (NULL until Week 2 NLP)  
    company\_name TEXT,  
    location TEXT,                 \-- City/region in Philippines  
    classification TEXT,           \-- ICT sub-category (e.g., 'Information Technology')  
    salary\_min FLOAT,              \-- Minimum salary (from API/scraper)  
    salary\_max FLOAT,              \-- Maximum salary (from API/scraper)  
    salary\_currency TEXT DEFAULT 'PHP',  
    track\_alignment TEXT,          \-- Assigned track after NLP: 'CS-IS', 'CS-GD', 'IT-WD', 'IT-NT' (NULL until Week 2)  
    source\_url TEXT,               \-- Direct link to the original posting (for audit/debug)  
    posted\_at TIMESTAMPTZ,         \-- When the employer published the listing  
    scraped\_at TIMESTAMPTZ DEFAULT NOW(), \-- When we collected it  
    description_embedding VECTOR(384), \-- SBERT embedding on description text (NULL until Week 2)  
    raw_json JSONB,                \-- Full original response for audit/re-processing  
    created\_at TIMESTAMPTZ DEFAULT NOW()  
);  
  
\-- Scope constraint: Only accept posts within the last 6 months  
\-- Enforced in scraper logic — do not ingest posted\_at older than 180 days.  
  
\-- B. Job Skills Junction Table (job\_skills)  
\-- Normalized many-to-many: one row per (job, skill) pair.  
\-- Populated in Week 2 NLP pipeline after skill extraction.  
CREATE TABLE job\_skills (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    job\_id UUID NOT NULL REFERENCES jobs\_raw(id) ON DELETE CASCADE,  
    skill\_id UUID NOT NULL REFERENCES skills\_library(id) ON DELETE CASCADE,  
    frequency INT DEFAULT 1,      \-- How many times this skill appeared in the description  
    UNIQUE(job\_id, skill\_id)  
);

#### **B. Curriculum Tables (Two-Level)**

Split into track-level (summary) and course-level (granular) to support both HNSW search per track AND per-course Bloom's distribution.

SQL  
\-- B1. Curriculum Tracks Table (curriculum\_tracks) — one row per specialization  
CREATE TABLE curriculum\_tracks (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    track\_code TEXT UNIQUE NOT NULL, \-- 'CS-IS', 'CS-GD', 'IT-WD', 'IT-NT'  
    track\_name TEXT NOT NULL,        \-- Full name e.g., 'Computer Science — Information Systems'  
    program\_type TEXT NOT NULL,      \-- 'BSCS' or 'BSIT' — controls thesis vs capstone logic (CHED CMO §5)  
    track\_summary TEXT,              \-- Aggregated description of the specialization  
    bloom\_distribution JSONB,        \-- Track-level Bloom heatmap data for dashboard  
    embedding VECTOR(384),           \-- SBERT on track\_summary  
    hybrid\_embedding VECTOR(448),   \-- \[SBERT (384) + Node2Vec (64)\] for Inference Engine  
    created\_at TIMESTAMPTZ DEFAULT NOW()  
);  
  
\-- B2. Curriculum Courses Table (curriculum\_courses) — one row per course outcome  
CREATE TABLE curriculum\_courses (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    track\_id UUID NOT NULL REFERENCES curriculum\_tracks(id) ON DELETE CASCADE,  
    course\_code TEXT,  
    course\_title TEXT,  
    course\_type TEXT DEFAULT 'professional', \-- 'core', 'professional', 'elective', 'GE' — controls modification flexibility  
    is\_ched\_protected BOOLEAN DEFAULT FALSE, \-- TRUE for CC101-CC106 (CHED-mandated, additions only)  
    units SMALLINT,                 \-- Credit units (for 146-unit floor validation)  
    hours\_per\_week SMALLINT,        \-- Contact hours (enables "reduce from X weeks" phrasing)  
    outcome\_text TEXT,              \-- Individual learning outcome text  
    bloom\_level SMALLINT CHECK (bloom\_level BETWEEN 1 AND 6), \-- Bloom's taxonomy level  
    embedding VECTOR(384),          \-- SBERT on outcome\_text  
    created\_at TIMESTAMPTZ DEFAULT NOW()  
);

#### **C. Processed Skills Library (skills\_library)**

This is where the **Inference Engine** will live. It links specific skills to their "Parents" (e.g., React $\\rightarrow$ JavaScript).

SQL  
CREATE TABLE skills\_library (  
    id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),  
    skill\_name TEXT UNIQUE NOT NULL,  
    parent\_skill\_id UUID REFERENCES skills\_library(id), \-- For the Inference Engine  
    category TEXT,                 \-- 'Frontend', 'Cloud', 'Data Science', etc.  
    industry\_demand\_weight FLOAT DEFAULT 0.0,           \-- For Adaptive Ranking  
    bloom\_level SMALLINT CHECK (bloom\_level BETWEEN 1 AND 6),  
    suggested\_topics JSONB,        \-- Pre-curated sub-topic list, e.g. {"topics": ["Dockerfile syntax", "Docker Compose", "K8s basics"]}  
    program\_outcomes TEXT[],       \-- CHED outcome codes, e.g. ARRAY['IT03', 'IT07'] — maps skill to CMO outcomes  
    embedding VECTOR(384),         \-- Raw SBERT  
    graph\_embedding VECTOR(64),    \-- GCN/Node2Vec output  
    hybrid\_embedding VECTOR(448)   \-- Combined vector for Inference Engine  
);

Add this SQL command at the very end of your schema script to enable the "lightning-fast" semantic search mentioned in your technical basis:

SQL  
\-- HNSW index for skills\_library hybrid embeddings (448-dim)  
CREATE INDEX ON skills\_library USING hnsw (hybrid\_embedding vector\_cosine\_ops)  
WITH (m \= 16, ef\_construction \= 64);  
  
\-- HNSW index for curriculum\_tracks hybrid embeddings (448-dim)  
CREATE INDEX ON curriculum\_tracks USING hnsw (hybrid\_embedding vector\_cosine\_ops)  
WITH (m \= 16, ef\_construction \= 64);  
  
\-- IVFFlat index for jobs\_raw description search (384-dim, high row count)  
\-- Use IVFFlat here (not HNSW) because jobs\_raw grows to 5,000+ rows — IVFFlat is better for large datasets.  
CREATE INDEX ON jobs\_raw USING ivfflat (description\_embedding vector\_cosine\_ops)  
WITH (lists \= 100);  
  
\-- GIN index on job\_skills for fast per-skill lookups  
CREATE INDEX idx\_job\_skills\_skill ON job\_skills (skill\_id);  
CREATE INDEX idx\_job\_skills\_job ON job\_skills (job\_id);  
  
\-- B-tree indexes for common filter queries  
CREATE INDEX idx\_jobs\_raw\_track ON jobs\_raw (track\_alignment);  
CREATE INDEX idx\_jobs\_raw\_posted ON jobs\_raw (posted\_at DESC);  
CREATE INDEX idx\_jobs\_raw\_source ON jobs\_raw (source);

### **4\. Why this schema matters for the Vision**

* **Auditability:** The `raw_json JSONB` in `jobs_raw` means if the model ever hallucinates a skill, we can go back and see the exact original text the scraper found. `source_url` and `posted_at` provide full provenance.  
* **Normalization:** `job_skills` is a junction table (many-to-many), not an array column. This enables `GROUP BY skill_id` queries for computing `industry_demand_weight` without array unnesting.  
* **Separation of Concerns:** `curriculum_tracks` holds one row per specialization with a summary embedding for HNSW search, while `curriculum_courses` holds one row per course outcome for granular Bloom's analysis. They are separate by design.  
* **Speed:** HNSW on `skills_library` and `curriculum_tracks` supports sub-millisecond nearest-neighbor searches. IVFFlat on `jobs_raw` handles the larger row count efficiently.  
* **Scalability:** The 6-month `posted_at` scope constraint keeps the table bounded. The `track_alignment` column allows fast per-track filtering without JOINs during inference.  
* **Inference Readiness:** `parent_skill_id` in `skills_library` allows the Inference Engine to traverse the skill graph (e.g., "Find all jobs that require React OR its parent JavaScript").  
* **CHED Compliance:** `is_ched_protected` prevents the system from suggesting removal of CHED-mandated core courses (CC101-CC106). `course_type` controls the modification flexibility hierarchy (elective > professional > core > GE). `program_type` ensures BSCS tracks follow thesis requirements while BSIT tracks follow capstone requirements. `program_outcomes` in `skills_library` enables every suggestion to cite the CHED outcomes it satisfies.  
* **LLM-Free Suggestions:** `suggested_topics` in `skills_library` stores pre-curated sub-topic breakdowns per skill (e.g., Docker → ["Dockerfile syntax", "Docker Compose", "K8s basics"]), enabling the system to generate specific topic-level recommendations without requiring external AI API calls.
